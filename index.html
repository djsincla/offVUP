Offline delivery

offVUP.

Turns VMware's VCF Upgrade Planner into a self-contained zip your customer can open by double-clicking — no web server, no internet, no admin rights.

View on GitHub Upstream project

The problem

VMware's planner is excellent, but it's built to be served over HTTP. You can't simply download a zip of it and send it to someone.

Each scenario page loads its data with a fetch() call. Browsers block fetch() on file:// URLs as a security rule — so an unzipped copy opens, renders its shell, and then shows No components available where the entire upgrade plan should be.

It fails quietly. The page looks like it loaded. Nothing tells you the data never arrived.

What the build changes

Three changes, and nothing else. No planning logic or content is touched.

ChangeWhy
Inline the scenario JSON Embeds each page's data directly and drops the fetch() call. This is the actual file:// fix.
Vendor mermaid Upstream loads the diagram library from a CDN, so every workflow diagram would break on an isolated network. Now it ships in the zip.
Remove the tracker Upstream embeds a hits.sh visitor-counter image. It leaks the viewer's IP and user-agent to a third-party server, and renders as a broken image offline.

How it works

One command. Python 3 and git are the only requirements.

  1. Clone and run

    Upstream is cloned automatically on first run and refreshed on later ones — you don't need to fetch it yourself.

  2. Pages are rewritten

    Data is inlined, mermaid is vendored locally, the tracker is stripped. The build aborts loudly if upstream changes its fetch() pattern or any external reference survives — it won't hand you a silently broken zip.

  3. Ship the zip

    You get dist/VCF-Upgrade-Planner.zip, about 3.6 MB. Small enough to email.

# build it
git clone https://github.com/djsincla/offVUP.git
cd offVUP
python3 build_offline.py

# -> dist/VCF-Upgrade-Planner.zip  (3.6 MB)

What your customer does

Unzip it. Double-click index.html. It opens in their browser and works. Windows, Mac, and Linux, from the same file.

Keep the folder together. The pages link to each other and share local libraries, so moving index.html out on its own will stop it working. This is called out in the README the customer receives.

Verified, not assumed

Every scenario page is rendered in headless Chrome from a real file:// URL and checked for the broken-state string. All 14 pass.

That check is what proves the fix: unmodified upstream renders No components available from file://, and the built copy renders the full component list.

Honest limitations

External links still need internet. The planner itself runs fully offline, but some pages link out to Broadcom documentation and the VMware Interoperability Matrix. Those links won't open on an isolated network. The planning tool works regardless.
